# 语义质量门槛配置化设计

## 背景

当前 `SemanticSearchService` 已经具备：

1. 余弦相似度计算
2. 字段权重
3. top-2 chunk 聚合
4. 保守的字段分级阈值过滤

上一轮已经通过“最低质量门槛 + 弱命中过滤”减少了占位 embedding 带来的假阳性。

但目前仍有两个问题：

1. 语义质量规则仍散落在 `SemanticSearchService` 内部，不够集中
2. 搜索页顶部链路说明还没有明确告诉用户：当前语义结果已经经过最低质量门槛过滤

## 目标

本轮把语义质量规则从 service 内的硬编码提升成单独的集中配置对象，并同步收紧搜索页顶部解释文案，使当前语义结果的可信度规则更：

1. 可维护
2. 可复用
3. 可解释

## 非目标

本次不做以下事项：

1. 不把质量门槛暴露到设置页
2. 不引入 provider 化运行时调参
3. 不调整产品面的开关数量
4. 不改结果项级别标签文案
5. 不接入真实 embedding 模型

## 方案选择

本次采用：

- **代码内集中配置对象 + 搜索页顶部链路说明补一句质量门控说明**

不做设置页可调参数，也不做全局动态配置系统。

## 设计原则

### 1. 策略必须集中，但不扩散产品复杂度

把：

1. 全局最低语义门槛
2. 字段级阈值
3. 搜索页说明所需文案

集中到单独对象中，但仍保持纯代码内配置。

### 2. SearchService 继续只负责执行，不负责定义策略常量

`SemanticSearchService` 应依赖一个明确策略对象，而不是直接持有多组私有 magic numbers。

### 3. 文案只做顶部链路说明补强

本轮只在 `SearchPage` 的顶部链路说明处补一句：

- 当前语义结果仅展示通过最低质量门槛的命中

不同时修改结果项级解释，避免一次改太散。

## 结构建议

新增一个单独文件，例如：

- `lib/features/search/application/semantic_quality_policy.dart`

职责：

1. 提供全局基线阈值
2. 提供字段级阈值解析
3. 提供统一说明文案

建议提供一个默认策略，例如：

- `SemanticQualityPolicy.conservativeMvp()`

## 代码行为

### 策略对象

策略对象至少包含：

1. `minimumSemanticScore`
2. `minimumThresholdFor(SemanticHitField field)`
3. 用于 UI 的简短说明文案

### Service 接入

`SemanticSearchService` 改为依赖策略对象进行判断：

1. chunk 得到 `weightedScore`
2. 通过策略对象读取该字段阈值
3. 不达标则过滤

### 搜索页说明文案

在 `SearchPage` 的顶部链路说明中，当语义信号参与统一结果时，补充一句更收紧的说明，例如：

- `当前语义结果仅展示通过最低质量门槛的命中。`

这句只需要出现在顶部链路说明里，不需要下沉到每一条结果。

## 测试设计

至少补齐以下测试：

1. 默认策略对 `title` / `noteBody` 返回不同阈值
2. `SemanticSearchService` 继续保留上一轮的高质量过滤行为
3. `SearchPage` 在语义信号参与结果时显示新的质量门控说明
4. 非语义参与场景下，不强行显示该说明

## 默认假设

1. 当前 MVP 阶段，语义质量策略仍以代码常量为主。
2. 配置化的目标是“集中管理”，不是“开放用户配置”。
3. 本轮重点是把质量规则与解释规则对齐，而不是继续大幅调阈值。

## 完成后的转向建议

这轮完成后，下一步更适合转向：

1. 语义召回解释细化（例如区分“高质量语义命中”和“辅助语义命中”）
2. 真实 embedding 能力接入前的索引质量可观测性

# 搜索页未刷新索引承接提示设计

## 背景

当前 `SearchSettingsPage` 已经支持：

1. 草稿态 diff 结果预期提示
2. 保存后底部行动条
3. `立即刷新`
4. `返回搜索`

但当用户选择 `返回搜索` 时，当前搜索页还无法承接这次状态：

- 设置已保存
- 这次保存会影响语义索引
- 当前尚未刷新索引

于是用户回到搜索页后，缺少一个明确提示来判断：

1. 当前语义结果是否已可信
2. 是否应该立刻刷新索引
3. 是否仍然只是沿用旧索引结果

## 目标

在 `SearchPage` 增加一张轻量承接卡片，用来提示：

1. 设置已保存，但语义结果尚未刷新索引
2. 可以直接在搜索页触发一次刷新索引
3. 刷新成功触发后，该提示消失

## 非目标

本次不做以下内容：

1. 不实现全局任务中心
2. 不做复杂跨页面消息队列
3. 不记录具体哪一个字段改动导致提示出现
4. 不增加“知道了”“稍后提醒我”等多余动作
5. 不改现有刷新结果反馈卡的职责

## 方案选择

本次采用：

- **搜索页顶部承接卡片**

不采用一次性 SnackBar，也不做全局持久提醒。

## 设计原则

### 1. 承接状态必须轻量

MVP 只需要表达一个事实：

- `设置已保存，但语义结果尚未刷新索引`

因此使用单独的轻量 state 即可，不引入复杂数据结构。

### 2. 状态只服务于“从设置页返回搜索页”的闭环

该状态不是通用通知中心，不承担别的用途。

### 3. 承接提示优先级高于普通结果解释

因为它直接影响当前搜索结果是否应被用户当作“最新语义结果”来看待。

## 状态模型

新增一个轻量 handoff state，例如：

- `searchPendingReindexHandoffProvider`

建议字段：

1. `visible`
2. `message`

MVP 阶段不需要记录更复杂的元信息。

## 状态写入时机

在 `SearchSettingsPage` 中：

1. 用户保存后
2. 本次保存包含 `reindexItems`
3. 用户点击 `返回搜索`

此时写入 handoff state。

## 搜索页展示规则

在 `SearchPage` 中，当 handoff state 可见时，显示一张承接卡片。

建议位置：

1. 搜索输入框之后
2. 搜索设置入口卡之后
3. 状态卡之前

这样可以在用户看到搜索结果前，先看到“当前语义结果尚未刷新”的提醒。

## 卡片文案

标题：

- `设置已保存，但语义结果还没刷新`

说明：

- `你刚保存了会影响语义索引的设置。刷新索引后，再判断当前语义结果会更准确。`

动作：

- `立即刷新索引`

## 交互规则

### 点击“立即刷新索引”

调用现有：

- `searchIndexControllerProvider.indexPendingAndRefresh()`

要求：

1. 成功触发后清除 handoff state
2. 复用已有 refresh session 与 refresh feedback 流程
3. 若触发失败，则 handoff state 保留，方便用户重试

### 提示何时消失

本次明确采用用户确认的规则：

- **点“立即刷新索引”后消失**

补充细化：

1. 触发成功后消失
2. 触发失败时不消失

## 结构建议

为避免把状态塞进已有 refresh feedback：

1. 在 `search_providers.dart` 新增独立 handoff provider / state
2. 在 `search_page.dart` 新增承接卡片 widget
3. 在 `search_settings_page.dart` 的 `返回搜索` 动作中写入 handoff state

## 测试设计

至少补齐以下测试：

1. `SearchPage` 在 handoff state 可见时显示承接卡片
2. 点击 `立即刷新索引` 会调用 `indexPendingAndRefresh()`
3. 成功触发后 handoff state 被清除
4. 失败时 handoff state 保留
5. handoff state 不存在时，搜索页不显示该卡片

## 默认假设

1. 这张卡片只针对“影响语义索引的设置刚刚保存但尚未刷新”这一类情况。
2. 普通即时生效的搜索范围改动不写入 handoff state。
3. 本轮重点是闭环补齐，不是扩展成全局通知系统。

## 完成后的转向建议

这轮完成后，“设置页 -> 搜索页 -> 刷新反馈”链路会基本收口。

此时更适合转向：

1. 真实语义质量提升
2. 索引构建效率 / 批量刷新体验

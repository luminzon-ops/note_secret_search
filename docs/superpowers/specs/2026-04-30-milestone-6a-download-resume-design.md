# Milestone 6-A 下载恢复设计

## 1. 目标

本设计聚焦 **Milestone 6：模型下载管理** 的第一执行切片：**断点续传 + 下载恢复**。

本轮目标不是“做完整个模型下载管理”，而是把当前已经具备的最小下载闭环推进到**可恢复、可继续、可解释**的状态，使模型下载从“能下载”提升为“可交付使用”。

本轮完成后，用户应能获得以下行为：

1. 模型下载被暂停后，再次开始时优先从已下载部分继续，而不是删除本地 partial 文件后重下。
2. 应用重启后，模型管理页能够恢复上一次下载任务的状态与进度语义，而不是把已有 partial 文件当成“未开始”。
3. 页面上的下载按钮与状态文案能够区分“首次下载”“继续下载”“失败重试”和“已安装”。
4. 当下载源不支持续传或本地 partial 文件失效时，系统会明确回退到全量重下，而不是静默产生错误语义。

本轮**不包含**：自动切源、源测速、下载后 benchmark、设备能力评级增强、目录扩充、签名校验。

---

## 2. 当前现状

当前仓库已经具备如下主链：

1. `assets/model_catalog/built_in_catalog.json` 提供目录驱动模型元数据。
2. `ModelManagementPage` 已能展示模型目录、手动选择下载源、下载任务状态、安装状态与运行时状态。
3. `ModelDownloadController.startDownload()` 已能创建任务、执行下载、做 checksum 校验、写入 registry，并在 embedding / llm 两类模型下载完成后触发 runtime 校验。
4. `ModelDownloadTask` 已有 `queued / downloading / paused / completed / failed` 状态，且已存储 `downloadedBytes`、`totalBytes`、`averageSpeed` 与 `resumable`。

但当前实现仍存在一个关键断层：

- `ModelDownloadService.download()` 在每次开始下载前都会删除已有目标文件；
- 页面虽然展示“暂停”和“断点续传支持”，但暂停后的再次下载仍会走全量重下；
- 应用重启后虽然数据库还保留任务记录，但页面并没有基于本地 partial 文件恢复“继续下载”的语义。

因此，当前状态可以描述为：

> 下载任务状态模型已经存在，但“断点续传/下载恢复”仍停留在展示层语义，尚未成为真实行为。

---

## 3. 范围界定

### 3.1 本轮明确纳入

本轮只做以下 4 项：

1. **暂停后恢复**
   - 保留本地 partial 文件。
   - 用户再次点击下载时，优先从已下载字节处继续。

2. **应用重启后恢复任务状态**
   - 页面重新打开时，根据数据库任务记录 + 本地文件长度判断该任务是否可恢复。
   - 可恢复任务在 UI 中显示为“继续下载”，而不是“开始下载”。

3. **partial 文件驱动的进度恢复**
   - 若本地已有部分文件，`downloadedBytes` 应以实际文件长度为准恢复。
   - `progress`、状态卡、字节数显示与按钮状态同步更新。

4. **恢复失败时的可解释回退**
   - 当源不支持 Range 请求、partial 文件损坏、或 partial 文件与任务记录不一致时，系统明确回退为全量下载。
   - UI 和任务状态需保留可解释性，不允许让用户看到“已暂停/可恢复”但实际重新下载全量且无说明的情况。

### 3.2 本轮明确排除

以下内容不在本轮范围内：

1. 自动切源 / 源测速 / 失败自动回退到备用源
2. benchmark 与设备能力分级
3. 模型目录扩充
4. 下载签名校验
5. 后台下载服务、系统通知、系统级持久任务
6. 多源 partial 文件迁移（例如从 A 源下了一半，改切到 B 源继续）

这些能力仍属于 Milestone 6，但应在后续子项目中单独设计与实现。

## 4. 推荐方案对比

### 方案 A：最小恢复闭环（推荐）

做法：

- 在现有 `ModelDownloadTask` 和 `ModelDownloadService` 上增量扩展；
- 恢复语义建立在 **同一 `modelId + sourceId` + 本地 partial 文件存在** 之上；
- 通过 HTTP Range + append 模式实现续传；
- 若源不支持续传，则回退到全量下载。

优点：

- 改动集中；
- 与现有页面和 repository 结构兼容；
- 风险可控，最容易在本轮做成可交付闭环。

缺点：

- 不解决多源切换下的复杂恢复；
- 不解决后台下载与系统级任务恢复。

### 方案 B：抽象完整下载会话层

做法：

- 新增独立的 download session/domain 层，把 partial 文件、HTTP 能力、恢复元数据、任务生命周期全部抽象出来；
- 页面与 controller 通过新会话层访问下载能力。

优点：

- 长期结构更完整；
- 便于后续接自动切源、后台下载与策略调度。

缺点：

- 当前范围下过重；
- 需要同时重构 provider、service、UI 语义，超出本轮最小切片目标。

### 方案 C：只做 UI 恢复语义，不做真实续传

做法：

- 页面可显示“继续下载”，但底层仍删除文件后重下。

优点：

- 代码改动最少。

缺点：

- 这是伪恢复，不满足用户对“断点续传”的真实期待；
- 会制造更强的语义欺骗。

### 结论

本轮应采用 **方案 A：最小恢复闭环**。

原因很直接：它以最小改动补齐真实能力，不把恢复语义做成假象，也不把范围扩展到完整下载平台重构。

## 5. 设计细节

### 5.1 恢复判定语义

一个任务被认为“可恢复”，需要同时满足：

1. 最新任务存在，且 `status` 为 `paused`、`queued`、`downloading` 或 `failed`；
2. 任务的 `modelId + sourceId` 与当前选中 source 一致；
3. 本地目标文件存在；
4. 本地目标文件大小大于 0；
5. 本地文件大小小于期望总大小，或者总大小未知但任务未完成。

只要其中任何一项不成立，就不应对用户展示“继续下载”语义。

### 5.2 下载服务行为

`ModelDownloadService` 将从“总是删除目标文件再下载”改为三段式：

1. **检查已有文件**
   - 获取目标文件路径；
   - 若文件不存在，走首次下载；
   - 若文件存在，读取当前长度作为 `existingBytes`。

2. **尝试续传**
   - 若 `existingBytes > 0`，发起带 `Range: bytes=<existingBytes>-` 的请求；
   - 若服务端返回支持续传的响应，则以 append 模式继续写入。

3. **回退全量下载**
   - 若服务端拒绝 Range 或恢复上下文无效，则删除旧 partial 文件并重新全量下载；
   - 同时在 controller 层把任务状态解释为“恢复失败，已重新开始下载”。

### 5.3 Controller 行为

`ModelDownloadController.startDownload()` 的职责将被细化为：

1. 判断当前任务是首次下载、恢复下载还是失败重试；
2. 调用服务前先同步 partial 文件长度到任务记录；
3. 下载过程中持续更新 `downloadedBytes / totalBytes / averageSpeed / resumable`；
4. 若恢复路径失败并降级为全量下载，确保数据库里的字节数同步重置；
5. 下载完成后继续沿用现有 checksum → registry → runtime 校验链。

本轮不新增复杂调度器；恢复仍由当前 controller 负责。

### 5.4 页面行为

`ModelManagementPage` 中的目录项按钮将遵循以下语义：

- **开始下载**：没有任务记录或没有 partial 文件
- **继续下载**：有可恢复任务或 partial 文件
- **重试下载**：任务失败且无可恢复 partial 文件
- **暂停**：当前任务正在下载
- **已安装**：registry 中已存在且文件有效

状态卡中将补充：

- 当前是否为恢复下载
- 当前来源
- 已恢复的下载字节数
- 若回退到全量下载，可展示统一说明文案

### 5.5 数据与状态边界

本轮不新增新的数据库表。继续使用现有 `ModelDownloadTask`，必要时只增量扩展字段或通过现有字段推导恢复语义。

优先顺序为：

1. 能用现有字段表达，就不新增字段；
2. 若必须新增字段，字段必须只服务于恢复语义，不引入未来自动切源的策略性负担。

目前优先考虑的做法是：

- 使用现有 `status`、`downloadedBytes`、`totalBytes`、`resumable`；
- 本地 partial 文件是否存在由 `ModelDownloadService` 读文件系统判断；
- 页面不直接碰文件系统，而通过 provider/Controller 暴露恢复状态。

## 6. 测试策略

本轮必须按 TDD 做，至少覆盖以下测试面：

### 6.1 Application 层

1. paused 任务重新开始时走恢复路径，而不是新建无关任务；
2. failed 任务若存在 partial 文件，可切换为继续下载语义；
3. 下载恢复过程中 `downloadedBytes` 会从已有文件长度起算；
4. 不支持续传时，任务能回退到全量下载并保持状态一致。

### 6.2 Infrastructure 层

1. 目标文件存在时不会无条件删除；
2. Range 模式下载时会以 append 写入；
3. 恢复失败时会清理旧 partial 文件并全量重下；
4. checksum 校验仍在最终文件上执行。

### 6.3 Presentation 层

1. 可恢复任务显示“继续下载”；
2. 首次下载显示“开始下载”；
3. 失败且无 partial 文件显示“重试下载”；
4. 状态卡能显示恢复后的进度与断点续传支持信息。

## 7. 风险与取舍

### 风险 1：HTTP 源不支持 Range

处理方式：

- 恢复失败时显式回退到全量下载；
- 不把“不支持 Range”的源错误标成“支持断点续传”。

### 风险 2：partial 文件存在但任务记录不可信

处理方式：

- 以文件系统长度为准修正 `downloadedBytes`；
- 若文件长度明显异常，则删除后全量重下。

### 风险 3：UI 语义与真实恢复行为脱节

处理方式：

- 页面不直接推断恢复状态；
- 统一由 application/controller 层提供“是否可恢复”的语义结果。

### 风险 4：把范围扩展到自动切源 / benchmark

处理方式：

- 本轮严格拒绝，把它们保留在 Milestone 6 后续子项目中。

---

## 8. 实施后的完成标准

当以下条件全部满足时，Milestone 6-A 可判定完成：

1. 暂停后的模型下载可以继续，而不是删文件重下；
2. 页面重开后可恢复显示“继续下载”而不是“开始下载”；
3. 已下载字节数、进度条和任务状态与 partial 文件一致；
4. 源不支持续传时，系统能明确回退到全量下载；
5. 所有新增应用层、基础设施层、页面层测试通过；
6. `flutter analyze` 不因本轮引入新的错误。

## 9. 下一步边界

Milestone 6-A 完成后，Milestone 6 的推荐后续顺序为：

1. **Milestone 6-B：自动切源 / 源测速 / 失败回退**
2. **Milestone 6-C：安装损坏探测 / 更强校验**
3. **Milestone 6-D：设备能力检测 / benchmark / 推荐档位**
4. **Milestone 6-E：目录扩充与更多真实可发布模型资源**

这样能保证每一轮都产生一个清晰、可验证、可交付的子闭环，而不是把整个 Milestone 6 搅成一个高风险大改动。

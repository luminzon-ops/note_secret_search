# 搜索结果质量解释增强设计

## 背景

当前 `SearchPage` 已经具备三类解释能力：

1. 无结果时的保守引导
2. 顶部“当前检索链路”摘要
3. 单条结果中的“排序依据”与“语义命中”说明

另外，上一轮已经补齐了“索引后自动刷新完成反馈”，用户现在可以知道：

1. 搜索状态已刷新
2. 当前结果是否发生变化

但用户仍然缺少一层更容易快速扫读的“结果质量解释”：

1. 当前前排结果主要是关键词主导、语义主导，还是双命中主导
2. 每条结果的命中类型能否被快速识别

当前虽然已有解释文本，但主要散落在不同卡片和结果项细节里。对 MVP 用户来说，理解成本仍偏高。

## 目标

在不改排序算法、不做复杂 explain panel、不增加新页面的前提下，提升 `SearchPage` 对结果质量的解释能力：

1. 增强顶部结果解释总览
2. 给每条结果增加轻量命中类型标签
3. 让用户能更快理解“为什么这些结果排在前面”

## 非目标

本次不做以下事项：

1. 不重写搜索融合 / 排序逻辑
2. 不展示复杂逐条 explain 面板
3. 不展示字段级标签串（如标题命中 / 标签命中 / 正文命中）
4. 不把设置页扩展为结果解释页
5. 不增加更多刷新 diff 说明

## 方案选择

用户已明确选择方案 C：

1. **结果解释总览增强**
2. **命中类型轻量标签**

并进一步确认轻量标签采用：

1. `双命中`
2. `关键词优先`
3. `语义辅助`

## 设计原则

### 1. 总览解释先于细节解释

顶部卡片先告诉用户“整体上发生了什么”，单条结果标签再告诉用户“这条结果属于哪一类命中类型”。

这样用户不需要先逐条阅读排序依据，就能快速建立整体判断。

### 2. 标签只做一层分类，不做字段级解释

轻量标签只解决“快速识别”，不承担所有解释职责。

字段级解释、强弱信号原因、语义命中细节，仍继续留在现有结果项细节文案中。

### 3. 使用现有 `matchSources` 语义，不新增复杂评分层

本次轻量标签完全基于现有 `SearchResultItem.matchSources` 推导：

1. 同时包含 `keyword` 与 `semantic` -> `双命中`
2. 仅包含 `keyword` -> `关键词优先`
3. 仅包含 `semantic` -> `语义辅助`

不引入新的优先级计算器，不复制排序逻辑。

## 顶部结果解释总览

当前已有 `_SearchPipelineSummaryCard`，本次继续在该卡片内增强解释，而不新增另一张平行卡片。

新增两部分内容：

### 1. 主导型解释文案

根据当前 unified results 的构成，输出一个更自然语言的总结。

建议仅统计**前排结果**，避免长列表后部干扰判断。保守起见，本次取 unified results 的前 5 条。

分类方式：

1. `双命中` 数量
2. `关键词优先` 数量
3. `语义辅助` 数量

文案规则：

#### 情况 A：前排结果以双命中为主

示例：

- `当前前排结果以双命中为主，关键词与语义信号共同参与排序。`

#### 情况 B：前排结果以关键词优先为主，且存在少量语义参与

示例：

- `当前前排结果以关键词命中为主，语义信号主要用于补充排序。`

#### 情况 C：前排结果几乎只有关键词结果

示例：

- `当前结果主要来自关键词检索，语义链路尚未明显参与前排排序。`

#### 情况 D：前排结果以语义辅助为主

示例：

- `当前前排结果更多依赖语义召回，适合继续检查命中摘要与上下文。`

### 2. 命中类型构成摘要

在“结果构成”下再补一行，例如：

- `前 5 条中：双命中 2 条，关键词优先 2 条，语义辅助 1 条。`

若结果不足 5 条，则按实际数量显示。

## 单条结果轻量标签

在每条结果卡的标题区或标题下方增加一行轻量标签，避免改成复杂大卡片。

标签映射规则固定为：

1. `keyword + semantic` -> `双命中`
2. `keyword only` -> `关键词优先`
3. `semantic only` -> `语义辅助`

表现形式保持轻量：

1. 使用 `Chip` / `Wrap`
2. 每条结果默认只显示一个命中类型标签
3. 不再叠加更多解释标签，避免噪音

## 结构建议

为避免 `search_page.dart` 继续膨胀，本次建议把这部分解释逻辑抽成小型 helper：

- `lib/features/search/presentation/search_result_explanation.dart`

职责建议：

1. 从 `SearchResultItem` 计算命中类型标签
2. 从 unified results 计算前排解释摘要
3. 提供轻量 view model / helper 函数给 `SearchPage` 使用

这样可以把文案和判定逻辑从页面 widget 中分离出来，便于测试。

## 测试设计

至少补齐以下测试：

1. **顶部总览：双命中主导**
   - 显示双命中主导文案
   - 显示命中类型构成摘要

2. **顶部总览：关键词主导且语义辅助参与**
   - 显示关键词主导文案

3. **顶部总览：仅关键词参与**
   - 显示关键词主导但语义未明显参与的文案

4. **顶部总览：语义辅助主导**
   - 显示语义主导文案

5. **单条结果显示 `双命中` 标签**

6. **单条结果显示 `关键词优先` 标签**

7. **单条结果显示 `语义辅助` 标签**

## 默认假设

1. 前排结果解释只统计 unified results 前 5 条。
2. “双命中为主”按前 5 条中数量最多的类型判断；并列时优先级为：双命中 > 关键词优先 > 语义辅助。
3. 标签文案使用用户已确认的固定中文文案，不引入更多可配置项。

## 完成后的转向建议

这轮完成后，`SearchPage` 的：

1. 刷新完成反馈
2. 检索链路总览
3. 结果质量解释
4. 单条结果快速识别

将形成一套比较完整的 MVP 搜索解释层。

届时不应继续在解释文案上做低收益打磨，更适合主动转向下一任务，例如：

1. 搜索结果与详情页之间的解释连续性增强
2. 搜索范围 / 索引设置变更后的结果预期提示

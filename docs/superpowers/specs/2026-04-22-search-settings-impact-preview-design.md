# 搜索设置结果预期提示设计

## 背景

当前 `SearchSettingsPage` 已经具备：

1. 检索范围控制
2. 语义索引设置
3. 索引状态与刷新动作
4. 语义链路 readiness / blocked guidance

但用户在修改设置时，仍然不容易立刻理解：

1. 哪些修改会立即影响搜索结果
2. 哪些修改需要重新索引后才会反映到语义结果
3. 当前是否已经有“需要重新索引”的新草稿变更

如果没有这层预期提示，用户就只能靠经验猜测：

- “我刚改了这个开关，到底是不是已经生效了？”
- “是不是现在就该回搜索页看看？”
- “还是要先重建索引？”

## 目标

在 `SearchSettingsPage` 中新增一层草稿态 diff 提示，帮助用户理解当前修改将如何影响结果：

1. 对比“已保存配置”与“当前草稿配置”
2. 动态提示哪些改动会立即影响结果
3. 动态提示哪些改动需要重新索引后生效
4. 结合当前索引状态给出保守建议

## 非目标

本次不做以下事项：

1. 不实现复杂任务系统
2. 不新增后台索引调度器
3. 不改搜索算法或融合逻辑
4. 不做逐字段 explain panel
5. 不做全局未保存草稿管理框架

## 方案选择

用户已明确要求采用 **方案 C：草稿态 diff 提示**。

因此本次不是只做静态说明卡，而是根据用户当前改动动态生成结果预期提示。

## 设计原则

### 1. 草稿态仅限当前页面

本次草稿态只服务于 `SearchSettingsPage`。离开页面后不保留复杂草稿会话，不引入全局 draft store。

### 2. 只区分两类影响

MVP 阶段只区分：

1. **立即影响结果**
2. **需要重新索引后生效**

不继续细分更多级别，以避免规则膨胀。

### 3. 提示基于 diff，而不是基于静态说明

只有当当前草稿与已保存配置不同，才显示更明确的预期提示；否则显示保守的默认说明。

## 草稿态范围

本次涉及两组配置：

### 1. SearchScope 草稿

字段包括：

1. `includeTitle`
2. `includeSecretNote`
3. `includePasswordField`
4. `includeUsername`
5. `includeUrl`
6. `includeTags`
7. `includeNoteBody`
8. `allowLocalEmbedding`
9. `allowExternalProviderAccess`

### 2. SearchIndexSettings 草稿

字段包括：

1. `autoIndexEnabled`
2. `includeSecretNotes`
3. `includeNoteBody`
4. `maxChunkLength`

## 影响规则

### 立即影响结果

以下草稿变更视为立即影响：

1. `includeTitle`
2. `includeSecretNote`
3. `includePasswordField`
4. `includeUsername`
5. `includeUrl`
6. `includeTags`
7. `includeNoteBody`
8. `allowLocalEmbedding`
9. `allowExternalProviderAccess`

原因：这些属于查询范围或检索链路启停，调整后搜索逻辑可直接读取新配置。

### 需要重新索引后生效

以下草稿变更视为需要重新索引：

1. `includeSecretNotes`
2. `includeNoteBody`
3. `maxChunkLength`

原因：这些属于索引构建内容和 chunk 切分策略，只有重新构建索引后语义结果才会更新。

### 不进入“结果预期提示”的字段

1. `autoIndexEnabled`

原因：它影响的是保存后是否自动索引，不直接改变当前搜索结果或索引内容规则。

## 页面行为

### 新增一张“结果预期提示卡”

建议放在 `SearchSettingsPage` 中索引状态卡之后、语义索引设置之前。

卡片回答三件事：

1. 当前草稿是否存在立即生效的改动
2. 当前草稿是否存在需要重新索引的改动
3. 当前建议用户下一步做什么

### 保存后的行动闭环

当用户点击：

1. `保存索引设置`
2. `保存检索范围`

并且**本次保存对应的草稿中包含“需要重新索引后生效”的改动**时，页面底部应显示一条持久行动条，补足“保存完成后下一步做什么”的闭环。

行动条文案：

- `设置已保存，语义结果需要刷新索引后更新。`

行动条动作：

1. `立即刷新`
2. `返回搜索`

### 行动条动作定义

#### 立即刷新

点击后调用现有：

- `searchIndexControllerProvider.indexPendingAndRefresh()`

要求：

1. 复用当前已有刷新 / 索引流程
2. 不新增独立后台任务系统
3. 成功触发后关闭行动条
4. 继续复用现有成功 / 失败反馈文案

#### 返回搜索

点击后：

1. 关闭行动条
2. 返回上一页；如果当前页不可 `pop`，则保守回到 `/`

该动作**不自动触发索引**，只允许用户带着“已保存但尚未刷新索引”的状态离开设置页。

### 行动条显示条件

只有在以下条件同时满足时才显示：

1. 用户刚完成一次保存
2. 本次保存包含 `reindexItems`
3. 当前不处于 refresh session 中

### 行动条隐藏条件

以下情况应隐藏或清除行动条：

1. 用户继续修改草稿
2. 用户点击 `立即刷新` 且成功进入刷新流程
3. 用户点击 `返回搜索`
4. 当前进入 refresh session
5. 本次保存只有“立即影响结果”的改动
6. 本次保存没有实际改动

### 情况 A：没有草稿改动

显示保守默认说明：

- `检索范围类设置会立即影响结果；索引内容类设置在你下次重建索引后生效。`

### 情况 B：只有立即影响的改动

显示：

- `你当前的草稿会立即影响搜索结果。保存后可以直接回到搜索页查看变化。`

并列出受影响项。

### 情况 C：只有需要重新索引的改动

显示：

- `你当前的草稿会影响语义索引内容。保存后需要重新索引，语义结果才会更新。`

若当前已有待索引项，则再补：

- `当前已有待索引内容，建议保存后直接刷新索引。`

### 情况 D：两类改动同时存在

显示：

- `你当前的草稿包含两类影响：部分改动会立即影响结果，部分改动需要重新索引后生效。`

并分区展示两类改动。

## 结构建议

为了避免 `SearchSettingsPage` 继续膨胀，本次建议抽一个 helper：

- `lib/features/search/presentation/search_settings_impact_preview.dart`

职责：

1. 比较已保存配置与草稿配置
2. 生成 impact preview view model
3. 输出用于 UI 的文案和分组

## 测试设计

至少补齐以下测试：

1. **无草稿改动时显示默认说明**
2. **存在立即影响改动时显示“立即影响结果”提示**
3. **存在需要重新索引改动时显示“需要重新索引”提示**
4. **两类改动同时存在时显示混合提示**
5. **有待索引项时补充更明确建议**
6. **保存后若存在需要重新索引改动，则显示行动条**
7. **点击“立即刷新”会调用 `indexPendingAndRefresh()`**
8. **点击“返回搜索”会离开设置页**
9. **只有立即生效改动保存后，不显示行动条**

## 默认假设

1. 草稿在当前页面内即可，不需要跨页面保存。
2. 用户的草稿改动一旦保存，provider 会更新为新的已保存配置。
3. 本轮重点是“结果预期提示”，不是“未保存草稿管理”。
4. 检索范围保存本身不天然意味着出现行动条，只有当本次保存对应 diff 内含 `reindexItems` 时才出现。

## 完成后的转向建议

这轮完成后，搜索相关体验将覆盖：

1. 列表解释
2. 刷新反馈
3. 详情页承接
4. 设置改动的结果预期

此时不应继续深挖解释文案，更适合转向下一任务，例如：

1. 真实语义质量提升
2. 索引性能 / 批量索引体验优化

# 详情页搜索解释轻量承接设计

## 背景

当前 `SearchPage` 已经完成：

1. 搜索结果刷新完成反馈
2. 顶部结果解释总览
3. 单条结果命中类型轻量标签

但用户从搜索结果进入详情页后，解释链路还不够统一。

虽然当前路由已经透传：

1. `query`
2. `source`
3. `context`

并且 `SecretDetailPage` / `NoteDetailPage` 已经能显示基础的：

- `来自搜索 · 关键词 + 语义`
- 查询词
- 命中上下文

但当前承接文案与列表页新增加的命中类型标签不一致。列表页已经使用：

1. `双命中`
2. `关键词优先`
3. `语义辅助`

因此搜索到详情的体验仍存在一层认知断点。

## 目标

在不改详情页主体结构、不增加复杂 explain panel 的前提下，为详情页补齐与列表页一致的搜索解释承接：

1. 仅在从搜索进入时显示顶部来源说明区块
2. 用与列表页一致的命中类型文案替代旧的来源文案
3. 继续展示查询词与命中上下文
4. 非搜索进入时不显示该区块

## 非目标

本次不做以下事项：

1. 不新增独立的复杂解释卡
2. 不为详情页增加更多字段级解释
3. 不增加搜索历史恢复或返回态恢复
4. 不重构详情页主体布局
5. 不引入新的 provider 或复杂状态管理

## 设计原则

### 1. 详情页只做轻量承接，不做新解释中心

详情页只承接用户从搜索结果页带过来的已有解释信息，不在详情页重新计算复杂解释。

### 2. 文案与列表页保持一致

详情页顶部的命中类型必须与列表页一致，避免用户在列表页看到 `双命中`，进入详情页却看到另一套术语。

### 3. 继续基于现有路由参数实现

本次不新增额外数据通道，直接基于已有：

1. `searchQuery`
2. `searchSource`
3. `searchContext`

完成承接。

## 文案映射

将当前详情页中的 `_searchSourceLabel` 映射规则调整为与列表页一致：

1. `keyword_semantic` -> `双命中`
2. `keyword` -> `关键词优先`
3. `semantic` -> `语义辅助`

因此顶部文案将从旧的：

- `来自搜索 · 关键词 + 语义`

变为：

- `来自搜索 · 双命中`

## 页面行为

### SecretDetailPage

当满足以下任一条件时，显示顶部来源说明区块：

1. `searchSource` 非空
2. `searchQuery` 非空
3. `searchContext` 非空

区块内容：

1. `来自搜索 · <命中类型>`
2. `查询词：...`（若存在）
3. `命中上下文：...`（若存在）

### NoteDetailPage

行为与 `SecretDetailPage` 完全一致。

### 非搜索进入

若三项都为空，则完全不显示该区块。

## 结构建议

当前两个详情页都已经有类似结构与 `_searchSourceLabel` 逻辑，适合做一次小型对齐：

1. 抽一个轻量 helper（可选）
2. 或至少统一两页的映射函数与显示条件

MVP 阶段优先保守：

- 如果抽 helper 会让改动更稳定，就抽
- 如果只是很小的映射改动，也可以先保持两页局部实现一致

## 测试设计

至少补齐以下测试：

1. `SecretDetailPage` 从搜索进入时显示：
   - `来自搜索 · 双命中`
   - 查询词
   - 命中上下文

2. `SecretDetailPage` 非搜索进入时不显示该区块

3. `NoteDetailPage` 从搜索进入时显示：
   - `来自搜索 · 关键词优先` 或 `语义辅助`
   - 查询词
   - 命中上下文

4. `NoteDetailPage` 非搜索进入时不显示该区块

## 默认假设

1. 详情页不重新计算命中类型，只依赖路由里的 `searchSource`。
2. 命中上下文仍直接显示传入的 `searchContext`，不做二次加工。
3. 本次目标是“解释连续性”，不是“详情解释增强”。

## 完成后的转向建议

这一轮完成后：

1. 列表页解释
2. 刷新反馈解释
3. 详情页来源承接

将形成比较完整的搜索解释闭环。

届时不建议继续细磨解释文案，更适合主动转向下一任务，例如：

1. 搜索范围 / 索引设置变更后的预期提示
2. 真实语义质量提升或索引体验增强

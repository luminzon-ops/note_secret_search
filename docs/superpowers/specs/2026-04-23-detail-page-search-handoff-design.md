# 详情页搜索承接卡片增强设计

## 背景

当前搜索页已经会在点击结果项后，把以下参数透传到详情页：

1. `query`
2. `source`
3. `context`

当前 `SecretDetailPage` 和 `NoteDetailPage` 也已经有一张基础的“来自搜索”卡片，但信息仍较粗糙：

1. 标题与命中方式混在同一行
2. `命中上下文` 文案偏技术描述
3. 没有把“为何打开这一条”的信息表达得足够直接

## 目标

在不新增复杂交互和不改搜索主链路的前提下，增强详情页顶部的搜索承接卡片，让用户进入详情页后能快速理解：

1. 这是从搜索进入的
2. 本次命中的方式是什么
3. 查询词是什么
4. 为什么这条内容被命中

## 非目标

本次不做：

1. 不把 SearchPage 的完整排序依据搬到详情页
2. 不新增搜索状态回查或跨页共享状态
3. 不重新计算命中理由
4. 不做新的导航入口
5. 不新增按钮、筛选器或操作

## 方案

采用：

- **增强现有顶部小卡片**

不新建新的组件体系，不改变详情页主内容布局，只增强已有卡片的文案结构。

## 信息来源

详情页卡片继续只依赖现有路由参数：

1. `searchQuery`
2. `searchSource`
3. `searchContext`

这意味着：

1. 不回查 SearchPage 状态
2. 不新增 provider 协调
3. 不在详情页重新推导搜索解释

## 卡片结构

当以下任一值存在时显示卡片：

1. `searchSource`
2. `searchQuery`
3. `searchContext`

卡片内容按顺序显示：

1. 标题：`来自搜索`
2. 命中方式：`命中方式：...`
3. 查询词：`查询词：...`
4. 命中说明：`命中说明：...`

仅显示有值的行。

## 命中方式映射

继续复用现有路由值，但把展示保持在详情页统一文案：

1. `keyword` -> `关键词优先`
2. `semantic` -> `语义辅助`
3. `keyword_semantic` -> `双命中`

如果 `searchSource` 为空，则不显示“命中方式”行。

## 命中说明文案

`searchContext` 不做拆分和重写，只更改展示标题：

- 从：`命中上下文`
- 改为：`命中说明`

原因：

1. 更接近用户理解
2. 不把详情页做成技术解释页
3. 直接复用搜索页已形成的解释文本

## 视觉约束

保持：

1. 仍为单个 `Card`
2. 放在详情页顶部
3. 使用简单纵向信息布局

不做：

1. 颜色等级编码
2. 额外图标分区
3. 可折叠说明
4. 芯片组或复杂摘要块

## 适用页面

本次同步增强：

1. `SecretDetailPage`
2. `NoteDetailPage`

要求两个页面的搜索承接结构保持一致。

## 测试设计

至少覆盖：

1. Secret detail 页显示增强后的搜索卡片
2. Note detail 页显示增强后的搜索卡片
3. `keyword / semantic / keyword_semantic` 映射正确
4. 没有搜索参数时不显示卡片

## 默认假设

1. 当前 detail 页的搜索承接已经是正确入口，无需改路由。
2. `searchContext` 已经足够表达“为什么命中”。
3. 详情页承接搜索解释的目标是帮助理解，而不是复刻搜索页的全部解释体系。

## 完成后的转向建议

这一轮收口后，更适合转向：

1. 搜索解释在详情页中的高亮承接（例如正文片段或字段高亮）
2. 搜索链路的结果可观测性

相比之下，不建议继续在详情页堆更多静态解释文案。

# 详情页命中位置承接设计

## 背景

当前搜索结果进入 `SecretDetailPage` 和 `NoteDetailPage` 后，详情页顶部已经可以展示：

1. 来自搜索
2. 命中方式
3. 查询词
4. 命中说明

但用户仍需要自己在详情页中寻找真正命中的位置。对 MVP 来说，当前最缺的是：

1. 让用户更快看到命中字段在哪
2. 尽量把页面移动到相关区域附近

## 目标

为 secret / note 详情页增加一层轻量的命中位置承接：

1. 根据 `searchContext` 推断命中字段
2. 对对应字段区域做轻量高亮
3. 在首帧完成后对该字段做一次性轻量滚动定位

## 非目标

本次不做：

1. 不做正文逐字高亮
2. 不做全文片段匹配
3. 不做多个命中点导航
4. 不做复杂动画
5. 不回查搜索页状态

## 方案

采用：

- **字段级高亮 + 一次性轻量滚动**

只对详情页中的字段块或字段行进行承接，不做文本内部的复杂定位。

## 信息来源

只使用已存在的详情页路由参数：

1. `searchContext`
2. `searchSource`
3. `searchQuery`

其中：

- 字段识别只依赖 `searchContext`

## 字段识别规则

新增一个轻量 helper，从 `searchContext` 的前缀识别目标字段。

### Secret detail

支持：

1. `标题：...` -> title
2. `账号：...` -> username
3. `网址：...` -> website
4. `标签：...` -> tags
5. `附注：...` 或 `备注：...` -> note

### Note detail

支持：

1. `标题：...` -> title
2. `摘要：...` -> summary
3. `正文：...` -> content
4. `标签：...` -> tags

如果无法识别：

1. 不高亮
2. 不自动滚动
3. 仍显示顶部搜索承接卡片

## 高亮形式

采用非常保守的样式：

1. 使用浅色背景或柔和描边
2. 保持现有布局不变
3. 不使用闪烁或复杂动画

每个可承接区域加统一视觉状态即可。

## 滚动策略

详情页加载后：

1. 若识别出目标字段
2. 在首帧渲染完成后仅滚动一次
3. 使用 `Scrollable.ensureVisible` 等保守方式让目标进入可见区域附近

约束：

1. 只触发一次
2. 目标不存在时不触发
3. 不做重复重试

## 结构建议

新增一个小 helper，例如放在 search/presentation 或 detail 页可复用位置，用于：

1. 解析 `searchContext`
2. 返回命中字段枚举

详情页内部根据枚举：

1. 决定哪个区域加高亮
2. 决定哪个区域挂 `GlobalKey` 参与滚动

## 页面范围

本次同步支持：

1. `SecretDetailPage`
2. `NoteDetailPage`

## 测试设计

至少覆盖：

1. helper 能识别 secret 相关字段
2. helper 能识别 note 相关字段
3. helper 对未知前缀返回未命中
4. secret detail 在字段可识别时展示高亮状态
5. note detail 在字段可识别时展示高亮状态
6. 无法识别字段时不误高亮

滚动本身不做像素级断言，测试重点放在：

1. 目标字段识别正确
2. 对应区域进入高亮状态

## 默认假设

1. `searchContext` 的前缀是当前最稳定的低成本信号。
2. MVP 阶段字段级承接已经足够，不需要全文定位系统。
3. 轻量滚动只作为体验增强，不应成为页面强依赖。

## 完成后的转向建议

这一轮收口后，更适合转向：

1. 搜索可观测性
2. 更真实的语义质量增强

不建议继续在详情页堆叠更多解释性文本。

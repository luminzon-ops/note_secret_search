## 2. 当前总体结论

### 2.1 当前整体进度判断

如果仅从“是否还停留在产品骨架阶段”来判断，旧版结论已经不再准确。

当前项目不只是基础安全入口、CRUD、关键词搜索和模型页骨架，而是已经额外完成了：

- **真实本地 embedding Android runtime 接入**；
- **本地 LLM Android runtime 接入**；
- **内置模型目录中的真实 GGUF 本地 LLM 候选接入**；
- **当前本地 LLM 选择 / 激活链路接入**；
- **正式 `/ai/chat` 页面与双 Tab 问答主链**；
- **私密问答 / 自由聊天 / 手动上下文 / 自动检索主链**；
- **AI 问答多会话持久化、恢复与切换**；
- **模型下载断点续传、应用重启后的下载恢复与自动切源主链**；
- **模型下载完成后对 embedding / LLM runtime 的真实状态回写**；
- **MiniCPM-V 4.6 多模态本地模型目录项、主模型 + 视觉投影双 artifact 下载聚合、registry artifact 持久化与 Dart / Android bridge 契约**。

因此，当前项目最准确的描述不再是”本地 AI 主线只完成到骨架 + 局部实现”，而应更新为：

> **Phase 1 的本地 AI 主链已基本完成。本地 LLM 文本生成（SmolLM2 360M / Qwen2.5 0.5B）已在 Huawei 真机上稳定运行，采样参数已从 Dart 层全链路可配置，token streaming 基础设施已接入；本地 embedding tokenizer 加载路径已修复；设备探测插件已就绪。但 MiniCPM-V 4.6 多模态 mtmd backend、模型运维、外部多 provider、同步架构与 AI 增强功能仍未完成。**

### 2.2 是否已经实现”本地部署 LLM + embedding 模型，并用于语义检索”

**结论：本地 LLM 文本生成已稳定闭环，本地 embedding 代码修复已完成。**

更准确地说：

- 本地 embedding 接入了 Android 原生 ONNX Runtime，tokenizer 加载路径已修复（§15）；
- 本地 LLM 接入了 Android 原生 GGUF / llama.cpp 风格 backend，通过反射 bypass + 显式 `n_predict` 修复了 Huawei Kirin 990 上的 native 崩溃（§13）；
- Flutter 侧 AI Chat / runtime provider / model management / persistence 主链均已存在；
- built-in catalog 已不再只有 embedding，而是额外接入了两个真实可下载的 GGUF 本地 LLM 候选（`Qwen2.5 0.5B Instruct Local` 与 `SmolLM2 360M Instruct Local`）；
- built-in catalog 还新增了 `MiniCPM-V 4.6 Q4_K_M Multimodal`，目录项按 `multimodal_llm` 类型描述主模型 GGUF 与 `mmproj` 视觉投影两个 required artifact；
- `/models` 页面现在已支持把已安装且 runtime-ready 的本地 LLM 设为“当前本地LLM”；
- 相关 focused tests 当前全部通过；
- 设备端已经完成 fresh install → 锁定页 → PIN 设置 → 解锁进入 `/vault` 的真实 smoke；
- `/models` 页面设备端 smoke 已补齐，已在模拟器上确认 `BGE Small 中文 Embedding`、真实下载源文案与模型说明；
- `/ai/chat` 页面设备端 smoke 已补齐，已在模拟器上确认 `AI 问答`、双 Tab、runtime banner、`前往模型管理` 与输入区文案真实可见。
- 华为真机上已确认 `files/models` 中存在 `qwen2_5_0_5b_instruct_q4_k_m.gguf` 与 `smollm2_360m_instruct_q4_k_m.gguf`，并且 `ai.active_llm_model_id` 指向 `smollm2_360m_instruct_q4_k_m`；
- 华为真机 `/ai/chat` 已确认显示 `本地 LLM 模型已经就绪：SmolLM2 360M Instruct Local`，但真实自由聊天生成在 native `librnllama_v8_2_fp16.so` 中触发 `SIGABRT`，因此本地 LLM 仍未达到设备端稳定交付状态。
- MiniCPM-V 4.6 的下载、artifact 持久化和 bridge payload 已有自动化证据，但 Android 侧目前仍是 `fallback_no_mtmd`，会返回 `runtime_unavailable`；真实图片推理仍需要接入 llama.cpp `mtmd` / vision backend 后才能闭环。

所以现在不能再说“完全未实现”，但也不能把它直接升级为“已完成 Phase 1 正式交付”。

### 2.3 当前开发方向是否仍然偏离主线

**旧版“明显偏离主线”的判断也已经部分过时。**

此前关于搜索 explainability、语义命中解释文案、详情页命中说明等工作，确实属于体验层优化；但当前仓库已不再只有这些 polish 项，而是已经补上了本地 embedding、本地 LLM、AI Chat 与会话持久化主链。

当前更准确的风险不再是”主线没做、只做 polish”，而是：

> **核心代码主链已完成；本地 LLM 文本生成在 Huawei 真机上的 native 崩溃已通过反射 bypass 修复；本地 embedding tokenizer 加载路径已修复；设备探测插件已就绪。剩余阻塞集中在 MiniCPM-V 4.6 mtmd backend、模型运维、设备能力约束 UI、多 provider 外部模型、安全硬化与同步架构。**

#### 13.1.2 修复 B — 真正的根因：绕开 `LlamaHelper.predict` 直接调用 `LlamaAndroid.launchCompletion` 并传入 `n_predict`

通过 `javap` 反编译 `llamacpp-kotlin-0.2.0-huawei-safe.aar` 内 `LlamaHelper.kt` 与 `LlamaContext.kt`，确认：

- `LlamaContext.completion(Map)` 支持 `n_predict / temperature / top_k / top_p / seed / stop / penalty_*` 等 20+ 采样参数。
- 但 `LlamaHelper.predict(String, Boolean)` 仅传入 `{prompt, emit_partial_completion}` 这 2 个字段就调用 `LlamaAndroid.launchCompletion`，导致 native llama.cpp 默认 `n_predict = -1` → **无界生成**。
- 在 Huawei Kirin 990 Cortex-A77 上，无界生成 + 缺省 sampling 在 `doCompletion+1284` 处分别触发 `SIGABRT (6)` / `SIGSEGV (11) SEGV_MAPERR` / `SIGBUS (7) BUS_ADRALN`（三种信号都在同一 JNI 偏移），fault addr 形似 ASCII 字节（`0x65737565722072 ≈ "r·euse"`），强烈指向 **JNI ABI 失配或 use-after-free**。

落地修复：在 `android/app/src/main/kotlin/com/example/note_secret_search/GgufLlamaCppBackend.kt` 的 `AndroidLlamaHelperClient.predict` 中**绕开 `LlamaHelper.predict`**，通过反射读取 AAR synthetic accessor（`LlamaHelper.access$getLlama` / `LlamaHelper.access$getCurrentContext$p`），直接调用 `LlamaAndroid.launchCompletion(contextId, params)`，其中 `params` 显式包含：

```kotlin
mapOf(
  "prompt" to prompt,
  "emit_partial_completion" to emitPartialCompletion,
  "n_predict" to maxTokens,                  // = config.maxOutputTokens (96)
  "temperature" to 0.7,
  "top_k" to 40,
  "top_p" to 0.9,
  "seed" to 42,
  "stop" to listOf("</s>", "<|im_end|>", "<|endoftext|>"),
)
```

完成后向 `sharedFlow` 主动 `tryEmit(LLMEvent.Done(text, 0, 0L))` 模拟 AAR 原有终态语义，保持 `awaitPredictionTerminalEvent` 等上层逻辑不变。如果反射失败则回退到原 `llamaHelper.predict`，确保更换 AAR 不会破坏 fallback。

## 15. 2026-05-18 本地 embedding 运行时修复

### 15.1 问题诊断

用户报告本地 LLM（SmolLM2 / Qwen2.5）在真机上可正常对话，但本地 embedding（语义检索）不可用。经诊断确认根因为 `OnnxEmbeddingRuntime.kt` 中两处 `context.assets.open()` 调用使用了 Flutter asset 路径，但 Android 原生 `AssetManager` 需要 `flutter_assets/` 前缀。

**详细链**：`bge-small-zh-v1.5` tokenizer JSON 文件是 Flutter asset（位于 APK 内 `assets/flutter_assets/assets/model_catalog/tokenizers/bge_small_zh_v1_5_tokenizer.json`），但 Kotlin 代码使用 `context.assets.open("assets/model_catalog/tokenizers/...")` 直接调用 Android `AssetManager`，缺少 `flutter_assets/` 前缀 → `FileNotFoundException` → `validateSpec()` 失败 → `loadTokenizer()` 失败 → `ensureModelReady()` 返回 `{status: "degraded"}` → Dart 侧 registry 写入 `enabled: false` → 模型被标记为不可用。

**对比**：GGUF LLM 模型使用文件系统直接访问（`File(modelPath)`），无 asset 依赖，因此不受此问题影响。

### 15.2 修复

**文件**：`android/app/src/main/kotlin/com/example/note_secret_search/OnnxEmbeddingRuntime.kt`

**两处改动**：
- 第 183 行（`loadTokenizer`）：`context.assets.open("flutter_assets/${spec.assetPath}")`
- 第 227 行（`validateSpec`）：`context.assets.open("flutter_assets/${spec.tokenizer.assetPath}")`

### 15.3 自动化验证

- `flutter analyze`：**No issues found**
- `./gradlew :app:testDebugUnitTest --tests "*OnnxEmbeddingModelSpecTest" --tests "*WordpieceEmbeddingTokenizerTest" --tests "*EmbeddingVectorPostProcessorTest"`：**全部通过**
- `flutter build apk --debug`：构建成功

### 15.4 设备侧状态

修复后的 APK 已安装到 Huawei `SPN-AL00`。因 `context.assets.open()` 在 APK 构建后即可正确解析 `flutter_assets/` 路径，tokenizer 加载链路已从代码层面打通。

设备端验证需在 Models 页面重新下载或重新激活 `bge_small_zh_v1_5`（模型文件已存在于 `files/models/bge_small_zh_v1_5/model.onnx`，但数据库 registry 在上轮真机调试时被清空）。用户可通过 Model Management → 找到 `BGE Small Chinese Embedding` → 点击下载（文件已落盘，仅需校验 + 重新写入 registry）完成激活。

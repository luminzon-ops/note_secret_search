# 第一阶段产品进度报告

## 1. 报告目的

本报告用于对照《产品开发实施方案》，审视当前项目在 **Phase 1 / MVP** 阶段的真实完成情况，判断：

1. 已完成了哪些基础能力；
2. 哪些能力仅完成了脚手架或部分实现；
3. 哪些关键里程碑尚未真正落地；
4. 当前工作重心是否偏离了既定主线；
5. 后续应如何调整开发优先级。

本报告结论基于以下两类证据：

- 产品规划基线：`产品开发实施方案.md`
- 当前项目代码与资源现状：`lib/`、`android/`、`assets/model_catalog/`、`test/`

---

## 2. 总体结论

### 2.1 当前整体进度判断

当前项目已经完成了 **MVP 的基础框架、基础安全入口、核心数据表、密码/笔记 CRUD、关键词搜索、语义索引链路骨架、模型目录与基础下载 UI** 等内容，说明项目已经从“空壳”推进到了“可运行的早期产品骨架”。

但如果严格对照《产品开发实施方案》的第一阶段目标：

> “先把**安全存储 + 检索 + 本地 AI + 模型下载管理 + 外部模型接入**跑通。”

那么当前项目 **还不能判定为已完成 Phase 1**，尤其以下三项仍明显未完成：

1. **本地 embedding 真实推理落地不足**；
2. **本地 LLM 能力尚未接入**；
3. **模型下载管理仍处于 MVP 骨架 / 半成品阶段**。

### 2.2 是否已经实现“本地部署 LLM + embedding 模型，并用于语义检索”？

**结论：没有完全实现。**

更准确地说：

- **本地 embedding 索引与检索流程骨架已存在**；
- **本地 embedding 模型管理与下载页面已存在**；
- 但 **真实 embedding 推理引擎尚未接入**，当前仍使用占位实现；
- **本地 LLM 完全未接入**，没有看到 llama.cpp / GGUF / RAG 问答链路；
- 因此目前还不能说已经实现了“本地部署 LLM + embedding 模型并用于语义检索”的闭环。

### 2.3 当前开发方向是否有偏移

**有一定偏移。**

最近一段开发已经在搜索 explainability、语义命中解释文案、详情页命中说明等方面做了不少完善，这些工作对产品体验有价值；但从《产品开发实施方案》的优先级来看，当前阶段更高优先级的主线本应是：

1. 本地 embedding 检索真正可用；
2. 本地 LLM 能力接入；
3. 模型下载与部署闭环；
4. 设备能力分级与 benchmark；
5. 外部模型接入框架。

因此，项目当前的关键任务不是继续深挖搜索文案层 polish，而是 **尽快回到本地模型部署与真实推理闭环主线**。

## 3. Phase 1 / MVP 总体完成度评估

按实施方案中 Milestone 1 ~ Milestone 9 进行评估：

| 里程碑 | 状态 | 结论 |
|---|---|---|
| Milestone 1：项目基础搭建 | **基本完成** | Flutter / Riverpod / 路由 / 主题 / 日志 / Android 原生桥 / SQLCipher 骨架都已建立 |
| Milestone 2：安全与数据基础 | **部分完成** | 生物识别、PIN、截屏保护、默认保险库、数据表、CRUD 已有实现；但加密实现仍偏 MVP 骨架 |
| Milestone 3：搜索基础能力 | **完成度较高** | 关键词搜索、范围配置、结果页、排序与融合能力已比较完整 |
| Milestone 4：本地 embedding 检索 | **部分完成** | chunk / 向量表 / 相似度计算 / 索引逻辑已做；但真实 embedding runtime 未接入 |
| Milestone 5：本地 LLM 能力 | **未完成** | 未见 llama.cpp、GGUF、RAG 问答、引用展示、LLM benchmark |
| Milestone 6：模型下载管理 | **部分完成但差距大** | 有内置目录、下载任务、安装记录、删除能力；但签名校验、断点续传、切源、测速、benchmark 未完成 |
| Milestone 7：外部模型接入 | **未完成** | 未见 OpenAI 兼容、Ollama、Anthropic、自定义 endpoint 配置实现 |
| Milestone 8：同步架构 | **基本未开始** | 数据表预留了 `sync_accounts`，但未见 SyncAdapter / WebDAV / 同步链路 |
| Milestone 9：AI 增强功能 | **未开始** | 未见敏感信息识别、安全风险提示、相似笔记建议、自然语言问答优化 |

综合判断：

- **基础能力完成较多**；
- **本地 AI 主线只完成到“骨架 + 局部实现”**；
- **距离第一阶段正式完成仍有关键缺口**。

   - `SharedPreferencesSecuritySettingsRepository.savePinMaterial()` 直接保存字符串

     - “当前为 MVP 骨架，后续会切换到真实 KDF / 安全包裹方案”

3. **设备根密钥仍是骨架实现**
   - `SecureKeyManager.kt` 注释直接说明未来才会接入 Keystore / StrongBox / 包裹方案
   - 当前仅在 SharedPreferences 中写入随机字符串作为数据库密码材料

4. **剪贴板自动清理只看到配置项，未看到实际执行链路**
   - 设置页展示 `clipboardClearSeconds`
   - 但未看到统一 clipboard 管理服务或自动清理实现

### 阶段判断

Milestone 2 已完成不少用户可见能力，但安全实现还偏 MVP 骨架。当前可以认为：

- **功能链路已具备**；
- **安全强度仍未达到方案目标上限**。

## 7. 关键偏差与风险判断

## 7.1 当前最关键偏差

当前项目最关键的偏差不是“没有做事”，而是：

> **已经在搜索结果体验层、命中说明层、术语统一层投入了不少开发，但 Phase 1 的核心差异化主线——本地 embedding、本地 LLM、模型下载运维——仍未真正闭环。**

这意味着：

- 搜索体验层已经开始变精细；
- 但底层真实 AI 能力还不扎实；
- 如果继续优先做文案 polish，会进一步拉大“产品表层完成度”和“核心能力完成度”的错位。

## 7.2 当前最重要风险

1. **误把占位 embedding 链路当作真实语义检索完成**
2. **误把模型目录 UI 当作模型部署能力完成**
3. **在没有 ONNX / llama.cpp / benchmark / 下载恢复能力前过度优化上层文案**
4. **安全能力在体验层可用，但加密实现仍过于保守，后续补安全会增加返工成本**

## 11. 最终结论

如果从“有没有做出产品骨架”这个角度看，当前项目已经明显走出了早期空白阶段，基础架构、数据层、CRUD、安全入口、搜索页、模型页都已经具备。

但如果从《产品开发实施方案》真正看重的差异化能力来判断：

> **本地优先、安全优先、真实本地 embedding、模型下载管理、后续本地 LLM 扩展**

那么当前项目仍处于：

> **基础骨架已成型，但核心 AI 能力尚未完成闭环的阶段。**

因此，第一阶段当前最准确的状态不是“快做完了”，而是：

> **基础能力完成较多，核心本地模型能力仍需集中攻坚。**

下一阶段必须把开发重心从 explainability polish 明确转回：

1. **真实本地 embedding 检索**
2. **模型下载与部署管理**
3. **设备分级与 benchmark**
4. **本地 LLM 最小闭环**

只有这几项补齐后，项目才可以被认为真正完成了《产品开发实施方案》定义下的 Phase 1 核心目标。

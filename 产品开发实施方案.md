# 移动端密码管理 / 私密备忘录软件开发方案

## 1. 文档目标

本文档面向开发者 AI 与研发团队，提供一份可直接落地执行的产品与技术开发方案。

产品定位：**密码管理器优先的私密信息管理应用**，兼顾私密备忘录、语义检索、本地模型运行、外部模型接入与多端同步扩展能力。

首发方向：**Android 优先**，兼容后续 iOS 扩展。

本次修订已纳入以下最终确认：
1. **不使用传统主密码**，改为仅依赖系统生物识别 / 应用 PIN 解锁。
2. **首版 UI 不直接暴露多保险库**，但架构保留 `vault_id` 与多保险库扩展能力。
3. **官方内置模型目录 + 模型下载部署能力** 作为重点功能建设。
4. **模型下载源支持国内镜像/备用源/局域网源切换**，重点优化下载速度与成功率。

---

## 2. 产品定位与设计原则

### 2.1 产品定位

这是一个以**密码管理**为核心、融合**私密备忘录**与**AI 语义检索**能力的移动端应用。

核心价值：
1. **安全存储**：安全保存账号密码、私密文本、敏感信息。
2. **快速找回**：通过关键词检索 + 语义检索快速定位所需内容。
3. **隐私优先**：默认启用极致隐私限制，优先本地推理，用户明确授权后才可调用外部模型。
4. **灵活智能**：支持本地模型、局域网模型、OpenAI 兼容接口、自定义 API。
5. **设备自适应**：根据设备性能推荐模型档位，自动避免高负载配置。
6. **模型可控**：官方模型目录、镜像源切换、断点续传、下载后兼容性检查形成完整模型运维体验。

### 2.2 目标用户

1. 对隐私敏感的普通用户
2. 有大量账号密码与私密笔记管理需求的重度用户
3. 希望使用本地 AI 检索能力的技术用户
4. 希望自定义模型与同步方案的高级用户

### 2.3 核心设计原则

1. **用户至上**：默认设置必须服务于隐私、安全、易用性。
2. **安全默认开启**：危险能力默认关闭，敏感外发必须显式确认。
3. **本地优先**：默认本地存储、本地加密、本地 embedding、本地语义检索。
4. **渐进增强**：低性能设备也应可运行基础功能，高性能设备可启用更强 AI 能力。
5. **模块化架构**：便于未来扩展自动填充、TOTP、Passkey、桌面端、浏览器扩展。
6. **下载体验优先**：模型下载功能是核心卖点，不能做成“能下但极慢、经常失败”的半成品。

## 3. MVP 与分阶段路线图

用户对密码管理深度的需求为“分阶段推进”，建议路线图如下。

### Phase 1：MVP（建议首个正式版本）

目标：先把**安全存储 + 检索 + 本地 AI + 模型下载管理 + 外部模型接入**跑通。

功能范围：
1. 应用锁
   - 系统生物识别
   - 可选应用 PIN（备用解锁方式）
   - Android 截屏保护
   - 最近任务缩略图保护
2. 密码管理基础能力
   - 新建密码条目
   - 字段：标题、账号、密码、网址、备注、标签、分类
   - 复制账号/密码
3. 私密文本备忘录
   - 第一版纯文本 / 轻量 Markdown
   - 标签、分类、收藏
4. 安全存储
   - 本地数据库加密（SQLCipher）
   - 敏感字段二次加密
   - API Key / 同步凭证加密
5. 检索能力
   - 标题/标签/内容关键词检索
   - embedding 语义检索
   - 用户可选是否把“密码等敏感字段”纳入检索索引
6. 本地模型能力
   - 本地 embedding 模型
   - 本地 LLM（先支持检索增强问答与相似内容解释）
7. 外部模型接入
   - OpenAI 兼容接口
   - Ollama
   - Anthropic
   - 自定义 Base URL + API Key + model name
   - 支持局域网服务
8. 模型下载管理（重点）
   - 最小版 / 省电版 / 高性能版
   - 展示模型大小、预估内存、预估速度
   - 官方内置模型目录
   - 国内镜像源优先 / 备用源切换
   - 断点续传、切换、删除
   - 下载后兼容性检查 + benchmark
9. 设备能力检测
   - RAM、CPU、存储、系统版本、电池/温控、可检测到的 GPU/NPU 信息
   - 首次 benchmark
   - 推荐模型档位
   - 禁止高负载模型
10. 用户自定义同步框架（先做架构，首版优先开放 WebDAV）

### Phase 2：增强版

1. 密码生成器
2. 密码强度检测
3. 重复密码检测
4. 安全审计面板
5. 同步扩展到 S3 / Dropbox / Google Drive / iCloud
6. 历史版本与回收站
7. 相似笔记合并建议
8. 敏感信息识别与风险提示
9. 基于私密内容的自然语言问答增强
10. 模型目录远程增量更新、灰度推荐与源质量评分

### Phase 3：高级版

1. TOTP / 2FA
2. Android Autofill
3. Passkey / FIDO2 预研与接入
4. 桌面端 / Web 辅助端
5. 浏览器扩展联动
6. 端到端同步优化、冲突处理完善
7. 多保险库 UI 能力正式开放

## 4. 功能模块拆解

## 4.1 账号与安全入口模块

### 功能点
1. 首次启动引导
2. 绑定系统生物识别
3. 可选启用应用 PIN
4. 设置自动锁定策略
5. 初始化设备密钥与本地默认保险库

### 已确认设计
- 不设置传统“主密码”，采用**仅依赖系统生物识别 / 应用 PIN** 的解锁模型。
- 生物识别为默认主解锁方式。
- PIN 为应用内可选的备用解锁方式，由用户在设置里主动开启。
- PIN 不作为长期主密钥，而是备用门禁。
- 加密根密钥由系统安全硬件/KeyStore 保护，应用不要求用户记忆额外主密码。
- 首版仅创建一个“默认保险库”，不在 UI 中直接暴露多保险库入口。
- 数据模型中保留 `vault_id`，为未来多保险库扩展留口。

---

## 4.2 应用锁与安全控制模块

### 已确认需求
1. 系统生物识别
2. 可选独立应用 PIN
3. Android 截屏保护

### 进一步建议纳入 MVP 的安全策略
1. 进入后台后可配置自动锁定（立即 / 30 秒 / 1 分钟 / 5 分钟）
2. 剪贴板清理（复制密码后 30 秒/60 秒自动清除）
3. 外部模型访问敏感内容二次确认
4. 敏感页面隐藏最近任务缩略图
5. PIN 失败次数限制与冷却时间

### Android 技术建议
- 使用 `FLAG_SECURE` 防截屏/录屏
- 敏感界面进入后台时遮罩
- 生物识别使用系统标准组件 `BiometricPrompt`
- PIN 失败计数持久化但不泄露实际 PIN 信息

## 4.6 AI 检索与问答模块

### 目标
让用户能够通过自然语言快速找回私密内容，同时保留对敏感字段的控制权。

### 能力拆分
1. **语义检索**
   - 输入自然语言
   - 将查询转成 embedding
   - 与本地索引做相似度匹配
   - 返回最相关条目

2. **检索增强问答（RAG）**
   - 先检索，再将命中文本摘要送入 LLM
   - 输出回答与来源引用

3. **相似内容建议**
   - 根据 embedding 聚类发现重复、近似内容

4. **敏感信息识别**
   - 检测身份证号、银行卡、密钥、恢复码等文本模式

5. **安全风险提示**
   - 重复密码
   - 弱密码
   - 明文敏感数据过多
   - 外部模型调用风险

### 检索内容选择机制
必须让用户可配置：
- 是否检索密码条目标题
- 是否检索账号字段
- 是否检索密码字段
- 是否检索备注字段
- 是否检索笔记正文
- 是否允许这些内容进入本地 embedding
- 是否允许这些内容被外部模型访问

这部分是产品差异化关键。

## 4.7 模型管理模块

### 模型档位建议

#### 最小版
- 目标：低端设备可运行
- embedding：小模型
- LLM：1B~2B 级别量化模型或关闭本地 LLM，仅保留 embedding 检索
- 特点：体积小、速度快、精度一般

#### 省电版
- 目标：大多数中端设备
- embedding：中小型模型
- LLM：2B~4B 量化模型
- 特点：平衡速度与效果

#### 高性能版
- 目标：高 RAM、高存储设备
- embedding：较高质量模型
- LLM：4B~8B 量化模型（视移动端落地能力）
- 特点：回答更强，但耗电和热量更高

### 下载管理能力
1. 展示模型大小
2. 展示最低 RAM 要求
3. 展示推荐设备等级
4. 展示预估检索速度 / 首次加载耗时
5. 支持断点续传
6. 下载完成校验 checksum
7. 模型切换/卸载
8. 下载后自动 benchmark
9. 若设备不达标则禁止启用高档模型
10. 支持官方内置模型目录
11. 支持多下载源 / 镜像源切换
12. 根据地区默认推荐更快的下载源
13. 支持失败自动切换备用源
14. 支持显示当前下载源、速度、剩余时间
15. 支持“仅 Wi‑Fi 下载”

### 官方模型目录设计建议
首版应内置一份官方模型目录（catalog），同时支持远程更新，至少包含：
- 模型 ID
- 名称
- 类型（embedding / llm）
- 档位（最小版 / 省电版 / 高性能版）
- 推荐用途
- 文件大小
- 量化信息
- 最低 RAM
- 推荐设备等级
- 下载地址列表（官方源 / 国内镜像源 / 备用源 / 局域网源）
- checksum
- 许可协议
- 发布时间
- 是否推荐
- 模型摘要说明
- 风险提示（发热/耗电/空间占用）

### 下载源策略建议
由于模型文件通常较大，下载体验是重点功能，建议：
1. 每个模型维护多个下载源
2. 支持官方源、国内镜像源、社区镜像源、局域网自托管源
3. 下载前自动测速或基于地区/历史表现推荐默认源
4. 下载失败时自动切换备用源
5. 对大文件采用分片下载与断点续传
6. 显示当前下载源、预计剩余时间、平均速度
7. 提供“仅 Wi‑Fi 下载”选项
8. 记录源成功率、平均耗时，作为下次排序依据

### 产品策略建议
- 对 LLM 与 embedding 分开管理
- 支持“仅下载 embedding 模型”模式
- 默认优先推荐本地 embedding，本地 LLM 视设备情况开启
- 官方目录由远程清单 + 本地内置兜底清单组成
- 清单文件建议使用签名 JSON，防止模型目录被篡改
- 国内用户默认优先可用镜像源，避免“下载几乎不可用”的糟糕体验

用户要求支持自定义 WebDAV / S3 / Dropbox / iCloud / Google Drive 等。

### 建议分阶段交付

#### Phase 1
- 统一同步抽象层
- 先实现 WebDAV
- 预留 S3 / Dropbox / Google Drive / iCloud adapter 接口

#### Phase 2
- S3
- Dropbox
- Google Drive
- iCloud（iOS 阶段重点）

### 同步对象建议
1. 加密数据库快照
2. 增量变更日志
3. 模型配置元数据（不一定同步模型文件本身）
4. 用户偏好设置

### 同步策略建议
- 本地优先，后台异步同步
- 冲突时按记录粒度合并，不建议简单整库覆盖
- 对 embedding 索引可选择：
  1. 不同步向量，仅同步原始数据后本地重建
  2. 同步索引元信息，不同步向量主体

### 建议
首版不要把“向量索引跨端同步”做复杂，优先采用：
- **同步原始加密数据**
- **各设备本地重建 embedding 索引**

这样更稳，也更符合设备差异化模型策略。

## 5. 安全架构设计

## 5.1 安全目标

1. 数据静态存储加密
2. 敏感字段最小暴露
3. 外部模型调用前明确授权
4. 解锁机制清晰
5. 同步过程不暴露明文
6. 无主密码前提下仍保持可恢复的本地安全边界

## 5.2 建议密钥体系

### 方案建议
采用**设备密钥 + 生物识别/PIN 访问控制**的分层密钥设计：

1. **Root Key / 根密钥**
   - 首次安装时随机生成
   - 不向用户展示
   - 保存在系统 KeyStore / StrongBox（若设备支持）保护的密钥容器中

2. **DEK（Data Encryption Key）**
   - 随机生成
   - 用于数据库字段加密 / 内容加密
   - 由 Root Key 包装后存储

3. **Biometric Gate**
   - 调用系统生物识别后，获取使用 KeyStore 中受保护密钥的权限
   - 成功后解封装 DEK，用于本次会话解锁

4. **App PIN Gate（可选）**
   - 用户主动开启后，PIN 不直接作为长期主密钥
   - 通过 PIN + 盐 + KDF（Argon2id）派生出辅助解锁密钥，用于解封装 DEK 的副本或二级包装密钥
   - 作为无生物识别 / 生物识别失败时的备用入口

### 加密建议
- KeyStore/StrongBox：优先使用系统安全硬件
- PIN KDF：Argon2id
- 对称加密：AES-256-GCM 或 XChaCha20-Poly1305
- 哈希：SHA-256 / SHA-512
- PIN 校验：不存储明文 PIN，不存储可逆验证信息

## 5.3 数据库存储策略

### 推荐
- 使用 **SQLCipher** 对 SQLite 整体加密
- 对高度敏感字段再做应用层二次加密

建议二次加密字段：
- username
- password
- website_url
- note content（可选）
- API Key
- sync credential
- provider 配置

### 为什么需要双层
1. SQLCipher 解决整库保护
2. 应用层字段加密有利于更细粒度控制与未来导出/同步安全设计

## 5.4 零知识策略

用户选择：**默认启用极致隐私限制，但允许关闭**。

因此建议：
1. 默认不开启外部模型敏感数据访问
2. 默认不将密码字段纳入外部语义处理
3. 默认本地 embedding 与本地 LLM 优先
4. 设置页必须有清晰隐私模式说明

建议提供三档隐私模式：
- 严格模式（默认）
- 平衡模式
- 自定义模式

## 7.2 数据流

### 新建密码条目流程
1. 用户输入内容
2. 应用层校验字段
3. 敏感字段应用层加密
4. 写入 SQLCipher
5. 触发索引任务
6. 根据用户配置决定是否生成 embedding
7. 更新搜索缓存

### 语义检索流程
1. 用户输入自然语言 query
2. 查询当前检索范围配置
3. 用当前 embedding 模型生成 query vector
4. 从本地向量索引中召回候选
5. 用规则融合关键词分数 + 向量分数
6. 返回结果
7. 如果用户启用问答，再将候选摘要送入 LLM 生成回答

### 外部模型问答流程
1. 检查隐私模式
2. 判断是否包含敏感字段
3. 弹出外发确认
4. 对 prompt 进行最小化裁剪
5. 调用 provider
6. 返回结果并记录审计日志

### 模型下载流程
1. 加载本地内置模型目录
2. 拉取远程目录增量更新
3. 按地区/测速推荐默认下载源
4. 用户选择模型档位与具体模型
5. 创建下载任务并持久化
6. 分片下载与断点续传
7. 下载完成后校验 checksum / 签名
8. 执行兼容性检查与 benchmark
9. 成功后写入 ModelRegistry 并允许启用

## 8. 向量检索设计

## 8.1 索引策略

### 文本切分建议
由于首版只做文本，可采用轻量 chunk 策略：
- 密码条目：每个字段组合成 1~2 个 chunk
- 笔记：按段落/长度切分
- chunk 长度建议 200~500 tokens 对齐近似文本长度控制

### 检索粒度
- SecretItem：title / username / note / url / password（可选）
- NoteItem：title + content

### 索引更新策略
1. 新建后异步生成 embedding
2. 编辑后仅重建受影响 chunk
3. 删除后清除向量
4. 切换 embedding 模型后后台重建索引

## 8.2 向量存储方案

移动端首版建议简化：
- 向量直接存 SQLCipher 表中（BLOB / JSON array）
- 相似度计算在本地原生模块实现

后续若数据量增大，可引入专用 ANN 方案。

### 原因
1. MVP 复杂度低
2. 用户数据规模一般不会大到必须上复杂向量数据库
3. 便于加密与备份

## 8.3 排序融合
建议综合评分：
- keyword_score * 0.35
- vector_score * 0.65

并允许按内容类型、分类、标签、时间排序。

### 保险库（Vault）说明
“保险库”可以理解为一个**独立加密的数据空间/容器**。它不是保险业务概念，而是密码管理器常见的信息隔离单位。

例如：
- 个人保险库：存个人账号、银行卡、私人笔记
- 工作保险库：存公司账号、工作系统凭证、工作笔记
- 家庭保险库：存家庭共享 Wi‑Fi、电视会员、设备账号

使用保险库的好处：
1. 方便未来做多套独立数据集合
2. 便于不同同步目标分开配置
3. 便于未来做导入导出、分享、冲突隔离
4. 对 AI 检索范围做更精细控制

首版可以只默认提供一个“默认保险库”，但架构上保留多保险库扩展能力。

## 18. 风险与应对

## 18.1 本地 LLM 移动端体验不稳定
**风险**：中低端机速度慢、发热严重。

**应对**：
1. 将 embedding 检索设为核心能力
2. 本地 LLM 作为增强功能
3. 自动档位推荐与禁用高负载模型

## 18.2 多 provider 协议差异大
**风险**：接入复杂、维护成本高。

**应对**：
1. 做统一 provider 抽象
2. 先支持最常用参数集合
3. 高级参数放到扩展配置

## 18.3 同步冲突复杂
**风险**：多设备同时编辑导致数据不一致。

**应对**：
1. 先做 WebDAV + 增量日志
2. 冲突记录可视化
3. 向量索引不直接同步

## 18.4 Flutter 与原生桥接成本
**风险**：模型推理、benchmark、下载管理依赖原生。

**应对**：
1. 一开始就按插件化设计
2. 核心推理逻辑放 Android 原生层
3. Flutter 负责业务编排与 UI

## 18.5 模型下载体验差
**风险**：模型大、网络慢、源失效导致核心卖点失败。

**应对**：
1. 官方目录维护多源
2. 国内镜像优先
3. 自动测速、自动切源、断点续传
4. 下载失败可无损恢复
5. 将下载体验作为专项质量指标

5. 外部模型接入必须围绕隐私确认设计
6. 模型下载与目录管理必须当作核心功能，而不是附属功能

## 22. 总结

这是一个具有明显差异化潜力的产品方向：
- 它不是普通密码管理器，而是**安全优先 + 本地 AI 检索 + 用户可控外部模型接入 + 模型下载管理**的隐私工具。
- 真正的竞争力不在“接了 LLM”，而在：
  1. 安全边界清晰
  2. 本地优先
  3. 检索范围可控
  4. 模型档位与设备自适应
  5. 模型目录与下载体验完善
  6. 同步与扩展架构提前设计

如果按本方案实施，建议优先把下面四件事做成：
1. **安全存储与解锁体验**
2. **本地 embedding 语义检索**
3. **模型管理与设备分级**
4. **高可用模型下载链路**

这四点一旦打稳，产品就具备了可用性、差异化与后续扩展基础。
